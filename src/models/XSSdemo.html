What is XSS?

XSS (Cross-Site Scripting) is a web security vulnerability that allows an attacker to inject malicious JavaScript into a website. That script is then executed in the victim’s browser as if it were trusted code from the site itself.

The result? Stolen cookies, hijacked sessions, fake forms, or users doing things they never intended to do. All because the application trusted input it really shouldn’t have. 


                    <script>
                      alert('XSS');
                    </script>

If a website displays user input without proper escaping, an attacker can inject JavaScript like this. When another user loads the page, the script runs in their browser.

XSS usually happens when user input is directly inserted into HTML, attributes, or JavaScript without validation or encoding. Common targets include comment sections, search fields, profile names, and error messages.

There are three main types: Stored XSS, Reflected XSS, and DOM-based XSS. Different mechanics, same bad idea: trusting user input.

How to defend against XSS

  • Always escape user input before rendering it in HTML
  • Use proper output encoding based on context (HTML, attribute, JS)
  • Never trust client-side validation alone
  • Use modern frameworks that auto-escape content
  • Apply Content Security Policy (CSP)

XSS is simple, common, and still everywhere. If your app accepts input and shows it back to users, XSS is already knocking.
